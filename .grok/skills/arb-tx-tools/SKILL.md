# arb-tx-tools ŌĆö Specialized Execution Wrappers for the AI Agent

This skill turns the Grok agent into a powerful on-chain debugger for the arbitrage bot by giving it three production-grade TypeScript-powered tools that wrap real execution environments.

## The Three Core Tools

### 1. Transaction Simulator
**Purpose**: Test any transaction (raw calldata or full arb routes) against *live mainnet Polygon state forks*.

**Two implementations provided**:
- **Local Anvil (Foundry)**: Full control ŌĆö impersonation, state overrides, deep EVM tracing, persistent fork for multiple experiments. Uses the same forge-std/Anvil tech the project's `sol/test/*Fork.t.sol` already uses.
- **Alchemy MCP (cloud)**: Zero local setup. Uses the connected `alchemy` MCP server tools (`simulateExecution`, `simulateAssetChanges`, `traceCall`, `debugTrace*`). Ideal for the agent ŌĆö fast "would this revert?" checks against real state.

**Key scripts**:
- `scripts/simulator.ts start-fork` ŌĆö spin up (or print command for) `anvil --fork-url $POLYGON_RPC_URL`
- `scripts/simulator.ts simulate --to 0x.. --data 0x..` ŌĆö run eth_call + auto-decode reverts
- `scripts/simulator.ts simulate-arb --calls '[...]'` ŌĆö high-level arb route tester on temp fork
- `scripts/simulator.ts alchemy` (or `guide`) ŌĆö exact `search_tool` + `use_tool` instructions for Alchemy MCP sims

**When to use**:
- Before submitting a candidate route
- Reproducing a production revert locally
- Testing calldata generated by `src/services/execution/builder.ts`
- Comparing off-chain math sim (`src/pipeline/simulator.ts`) vs on-chain reality

### 2. ABICoder Tool
**Purpose**: Automatically decode transaction inputs, event logs, and especially revert data using *every ABI the project actually uses* (ArbExecutor custom errors, all swap interfaces, Balancer/Aave flash loan ABIs, etc.).

**Why this matters**: Most reverts in the bot are custom errors like `ExternalCallFailed(index, target, reason)`, `InsufficientProfit`, `FlashLoanOnly`, `InvalidPoolCaller`, or nested swap reverts. Generic "execution reverted" tells you nothing ŌĆö this tool tells you *exactly* which check failed and with what values.

**Script**: `scripts/abicoder.ts`
- `decode-input --data 0x...`
- `decode-revert --data 0x...`  ŌåÉ most important for trades
- `decode-receipt --hash 0x...`  ŌåÉ decodes every log + detects status=reverted
- `decode-tx --hash 0x...`
- `4byte 0x12345678`

It imports directly from `src/services/execution/calldata/abis.ts` + `hyperindex/abis/*.json` so it is always in sync with the real contracts.

**Typical workflow**:
1. Trade fails or `dryrun.ts` reports revert
2. Copy the `revertData` or tx hash
3. Run abicoder ŌåÆ get `ExternalCallFailed(2, 0xUniswapV3Pool, 0x08c379a0...)`
4. Feed that inner bytes back into abicoder ŌåÆ root cause (e.g. "STF" or "Too little received")

### 3. Log Tailer
**Purpose**: Feed the AI the *actual runtime TypeScript stack traces and errors* the bot is emitting right now.

The bot uses pino. When running with TUI (`--tui`) or file logging, everything goes to `data/runner.log` (structured JSON). The tailer parses it, surfaces stacks, and filters for the exact classes of errors the AI needs to fix instantly:

- Unhandled promise rejections
- JSON-RPC rate limit / 429 / timeout / connection reset
- HyperIndex / Hasura / effect failures
- Execution submission errors, nonce issues, gas estimation explosions
- Any ERROR/FATAL with stack

**Script**: `scripts/log-tailer.ts`
- `--last 150 --errors-only`
- `--follow --filter "unhandled|rejection|rate|JSON-RPC"`
- `--search "flashloan" --last 300`

Also prints human-friendly formatted output (timestamps + abbreviated stacks) so the AI can act immediately.

**Pro tip for unhandled rejections**: Run the bot as `node --trace-warnings --trace-uncaught src/cli/main.ts ...` so native stacks also land in logs or stderr (which you can `tail` separately).

**Automatic**: This skill description contains the trigger keywords. When the user (or your own reasoning) mentions "simulate this tx", "why did the arb revert", "decode the revert data", "tail the logs for the rate limit error", "start an anvil fork", etc., you should activate the skill.

**Explicit**:
- `/arb-tx-tools`
- `/skills arb-tx-tools`

Inside the skill you will see the exact command lines to run via your `run_terminal_command` tool.

## Recommended Debugging Loop (the power of this skill)

1. **Observe failure** ŌĆö either from TUI, user report, or `log-tailer.ts --errors-only`
2. **Reproduce exactly** ŌĆö use `simulator.ts simulate` (or Alchemy MCP path) with the real calldata that was going to be sent.
3. **Decode the revert** ŌĆö pipe the revert bytes straight into `abicoder.ts decode-revert`
4. **Fix root cause** (in builder, math, assessment, calldata, or contract)
5. **Verify fix** ŌĆö re-run the simulator on a fresh fork or via MCP; confirm the tx now succeeds and profit numbers match the TS pipeline sim.
6. **(Optional)** tail logs live during a real execution pass to catch any new runtime TS issues.

## MCP Tool Integration Notes

Because the `alchemy` MCP server is connected in this environment, the Transaction Simulator skill *explicitly* teaches you the two-step pattern:

```text
search_tool({ query: "alchemy simulateExecution traceCall" })
use_tool({ tool_name: "alchemy__simulateExecution", tool_input: { ... } })
```

This is the "mcp-tool" wrapper part ŌĆö the skill doesn't re-implement the transport; it gives you the precise incantation so you can treat Alchemy's simulation endpoints as first-class tools available to you as the agent.

Local Anvil is the escape hatch when you need state overrides, persistent impersonated accounts, or to debug the exact same EVM the Solidity fork tests use.
