My Recommendation: Targeted HyperIndex Ingestion

Framed around your exact problem: limited RPC endpoints → poor hydration → stale state → missed profit

The core insight

RPC polling is the bottleneck for profitable arbitrage. HyperSync gives you ~2000x faster historical data, and event-driven updates eliminate RPC polling entirely for the hot path.

Current: RPC-heavy
  • getReserves() every poll
  • slot0() every poll
  • liquidity() every poll
  • decimals() on discovery
  • balances(i) for Curve
  • getPoolTokens() for Balancer
~10-20 RPC calls per pool per refresh
Proposed: Event-driven
  • Sync event → reserves ✓
  • Swap event → sqrtPriceX96 + liq ✓
  • Decimals via createEffect (cached)
  • Curve A/fee via createEffect (cached)
  • Balancer via createEffect (cached)
~0-2 RPC calls per pool (one-time metadata)

What this looks like

// config.yaml — contracts + chain config
// schema.graphql — PoolState entities

// Strategy engine (unchanged)
pass_loop reads PoolState from DB instead of in-memory Map

Trade-offs

Pros

  • Zero RPC polling for live pool state
  • Auto-dedup + caching + rate limiting via createEffect
  • Reorg-safe (HyperIndex handles it)
  • Typed handlers (no manual decoding)
  • Strategy engine completely unchanged

Cons

  • Must adopt HyperIndex project structure (~5 new files)
  • DB-based state sharing (instead of in-memory Map)
  • Learning curve for config.yaml + schema.graphql

This is not a full rewrite — ~300 lines of new code, ~500 lines removed. The strategy, execution, mempool, CLI, and config stay untouched.
